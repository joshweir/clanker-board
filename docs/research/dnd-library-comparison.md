# React Drag-and-Drop Library Comparison for Kanban Board (2026)

> Research date: 2026-07-18. New doc location - no existing `docs/research/` convention
> was found in this repo prior to this file (only `.wayfinder/tracker.md` for planning
> docs, and top-level `CLAUDE.md`/`AGENTS.md`). Created here as the obvious home for
> future research write-ups.

## Requirements recap

- Cross-column card drag, within-column reorder, column-header reorder.
- Snap-on-hover drop indicator (visual "it will land here" feedback).
- **Fractional-rank model**: the library must only report source index /
  destination index / target index at drag-end. The app computes a fractional
  rank (LexoRank-style) between neighbors and persists it. A library that
  insists on owning and re-sorting an `items` array internally is a worse fit
  than one that is purely event/hook-based.
- React 19 + Vite + TanStack Router SPA, desktop-first, accessibility matters.

## @atlaskit/pragmatic-drag-and-drop

**Maintenance.** Core package at `2.0.1` (published 2026-06-17); sibling packages (`react-drop-indicator@4.1.1`, `react-accessibility@3.1.4`, `flourish@3.0.2`) published as recently as 2026-07-17 - the day before this research. ([npm registry](https://registry.npmjs.org/@atlaskit/pragmatic-drag-and-drop)) The GitHub repo ([atlassian/pragmatic-drag-and-drop](https://github.com/atlassian/pragmatic-drag-and-drop), 12,705 stars) is an explicit **one-way mirror** of Atlassian's internal monorepo - "not intended to accept code contributions," commits titled "Synchronise latest changes" landing every 1-4 days. No GitHub Releases/tags exist; versioning lives in per-package `CHANGELOG.md` + npm only. Issue ratio (94 open / 86 closed) suggests active triage, not abandonment.

**React 19.** The core (`pragmatic-drag-and-drop`, `-hitbox`, `-live-region`) is framework-agnostic vanilla TS with **no React peerDependency at all**. The React-specific opt-in adapters: `react-drop-indicator@4.1.1` and `react-accessibility@3.1.4` both declare `react: "^18.2.0 || ^19.0.0"`. However `react-beautiful-dnd-migration@3.1.1` (for teams migrating off rbd) is still capped at `react: "^18.2.0"` - React 19 unsupported there. [GitHub issue #181 "Support React 19"](https://github.com/atlassian/pragmatic-drag-and-drop/issues/181) (opened 2025-01-14) remains open, though most packages have since shipped React 19 peer ranges.

**Accessibility.** No automatic keyboard reordering. Official guidance is explicit: _"The core package does not enable accessible controls automatically, as there is no one pattern that works well for all situations"_ ([accessibility guidelines](https://atlassian.design/components/pragmatic-drag-and-drop/accessibility-guidelines)). They actively discourage arrow-key dragging and recommend an **action-menu pattern** (a per-card menu offering "move to top" / "move to column X"), aided by the opt-in `react-accessibility` package's `DragHandleButton`. Screen-reader announcements require manually calling `announce()` from `pragmatic-drag-and-drop-live-region` - confirmed via the official board example, which calls `liveRegion.announce(...)` inside a `useEffect` after each reorder.

**Touch/pointer.** Works without configuration, but real users report reliability problems: [discussion #93](https://github.com/atlassian/pragmatic-drag-and-drop/discussions/93) describes drops registering only ~10% of the time on touch with a longer press-and-hold delay than react-beautiful-dnd had, and no documented way to shorten it.

**Bundle size.** Because functionality is split across granular subpath imports, package-level numbers understate real cost: root `pragmatic-drag-and-drop@2.0.1` is 194B/164B gzip, `-hitbox@2.0.0` 194B/162B gzip, `-live-region@2.0.0` 805B/483B gzip ([bundlephobia](https://bundlephobia.com/package/@atlaskit/pragmatic-drag-and-drop)). `react-drop-indicator`/`flourish` builds repeatedly failed on bundlephobia during this research; a secondary (unverified) figure of ~4.7kB gzip for the practical "core + adapters" footprint appears in third-party write-ups. Measure per actually-imported subpath before trusting a single headline number.

**Fit for kanban / fractional-rank model.** Confirmed against the official board example ([`board.tsx`](https://github.com/atlassian/pragmatic-drag-and-drop/blob/main/packages/documentation/examples/board.tsx), [`card.tsx`](https://github.com/atlassian/pragmatic-drag-and-drop/blob/main/packages/documentation/examples/pieces/board/card.tsx)): a single `monitorForElements({ onDrop({ source, location }) {...} })` reports `source.data` (whatever domain data you attached) and `location.current.dropTargets` (ordered innermost-first: card, then column). The app calls `extractClosestEdge(target.data)` (from `-hitbox/closest-edge`) and the pure function `getReorderDestinationIndex({ startIndex, indexOfTarget, closestEdgeOfTarget, axis })` to get a final index - then the **app's own** `reorderColumn`/`reorderCard`/`moveCard` callbacks do the actual state update via `setData()`. The library never owns or mutates any array; there is no context/provider that requires a live ordered id array to render. This is the cleanest fit for a fractional-rank model of the four evaluated - swapping array-splice logic for "compute a rank between neighbors at `destinationIndex-1` and `destinationIndex`" requires touching only your own reorder callbacks. Snap-on-hover is a genuine out-of-the-box feature: `@atlaskit/pragmatic-drag-and-drop-react-drop-indicator`'s `DropIndicator` component is consumed directly in the official card example, driven by `closestEdge` state.

**License.** Apache-2.0, confirmed via [LICENSE file](https://github.com/atlassian/pragmatic-drag-and-drop/blob/main/LICENSE) and every checked package.json (GitHub's own metadata API misreports this as "Other/NOASSERTION" due to an auto-detection failure - the LICENSE text and npm registry `license` field are authoritative).

**Gotchas.** Touch drop reliability ([discussion #93](https://github.com/atlassian/pragmatic-drag-and-drop/discussions/93), [issue #12](https://github.com/atlassian/pragmatic-drag-and-drop/issues/12)); `onDrop` occasionally reporting an incorrect source/target element in complex nested layouts ([issue #179](https://github.com/atlassian/pragmatic-drag-and-drop/issues/179)); uneven React 19 rollout across adapter packages; zero built-in accessibility (deliberate design choice, not a bug, but real integration work); repo is read-only so bug fixes depend entirely on Atlassian's internal roadmap with no external PR path.

## dnd-kit (core + sortable)

**Maintenance - the key risk.** The classic API most kanban tutorials use is **frozen**: `@dnd-kit/core@6.3.1` and `@dnd-kit/sortable@10.0.0` both last published **2024-12-04/05** ([registry](https://registry.npmjs.org/@dnd-kit/core)) - no releases in ~19 months as of this research. The repo itself is not dead: it has undergone a full framework-agnostic rewrite (`@dnd-kit/abstract`, `@dnd-kit/dom`, `@dnd-kit/react`, plus Vue/Svelte/Solid adapters), first released 2026-06-11 with betas as recent as 2026-07-13 ([GitHub releases](https://github.com/clauderic/dnd-kit/releases)). The maintainer confirmed directly in [issue #1830](https://github.com/clauderic/dnd-kit/issues/1830) (2025-11-04): _"All development efforts are currently under the `experimental` branch which will soon replace the `main` branch... there may be breaking API changes before the `1.0.0` release."_ The classic packages now effectively live on a legacy branch/doc tree while the replacement is still pre-1.0.

**React 19.** `@dnd-kit/core@6.3.1` peer: `react: ">=16.8.0"` (open range, technically satisfied). Real breakage has been reported under React 19 - TS `JSX` namespace errors ([issue #1559](https://github.com/clauderic/dnd-kit/issues/1559)) and general "Support React 19 & Next.js 15" complaints needing `--legacy-peer-deps` ([issue #1511](https://github.com/clauderic/dnd-kit/issues/1511)). Both were closed in 2026-02 with the identical maintainer response: _"This issue was opened against a previous version of dnd-kit. The library has since undergone a major rewrite... Migration guide: https://dndkit.com/react/migration."_ - i.e. **the classic packages will not receive further React-19 fixes**; only the pre-1.0 `@dnd-kit/react` (peer `^18.0.0 || ^19.0.0`) is the React-19-certified line.

**Accessibility.** Genuinely strong out of the box, per [official a11y docs](https://docs.dndkit.com/guides/accessibility): a built-in `KeyboardSensor` (Enter/Space to grab, Escape to cancel, arrow keys move) and an off-screen live-region element announcing drag start/over/end automatically. `useDraggable` sets `tabindex="0"`, `role="button"`, `aria-roledescription="draggable"` by default.

**Touch/pointer.** Out of the box: _"By default, DndContext employs the Pointer and Keyboard sensors"_ ([sensors docs](https://docs.dndkit.com/api-documentation/sensors)) - `PointerSensor` covers mouse, pen, and touch via the Pointer Events API. Caveat: it can't `preventDefault()` native touch scrolling, so `touch-action: none` CSS or a dedicated `TouchSensor` is recommended for fine-grained control.

**Bundle size.** `@dnd-kit/core@6.3.1`: 42.68KB min / 13.90KB gzip. `@dnd-kit/sortable@10.0.0`: 8.96KB min / 3.58KB gzip ([bundlephobia](https://bundlephobia.com/package/@dnd-kit/core)). Both numbers are shallow per-package figures that don't fold in transitive deps (`@dnd-kit/accessibility`, `@dnd-kit/utilities`), so real combined footprint runs somewhat above ~17.5KB gzip.

**Fit for kanban / fractional-rank model.** Official "Multiple Containers" story ([source](https://github.com/clauderic/dnd-kit/blob/master/stories/2%20-%20Presets/Sortable/MultipleContainers.tsx)) confirms the reorder computation itself is index/event-based - `onDragEnd({active, over})` gives ids only; the app resolves index via `.indexOf()` and calls `arrayMove()`. But there's a real nuance: [`SortableContext` docs](https://docs.dndkit.com/presets/sortable/sortable-context) state it _"requires... a sorted array of the unique identifiers"_ and warns _"the `items` prop passed to `SortableContext` be sorted in the same order in which the items are rendered, otherwise you may see unexpected results"_ - this array drives `SortableContext`'s internal transform/animation math, so it's a hard rendering requirement, not optional. For a fractional-rank design this is workable but not free: you can't skip maintaining an ordered id array as a prop, though it can be **derived** each render from your rank-sorted domain objects rather than being the canonical persisted store. This makes dnd-kit meaningfully more array-coupled than pragmatic-drag-and-drop, which has no equivalent provider requirement. No shipped drop-indicator/insertion-line component exists - `DragOverlay` only renders a floating clone of the dragged item; the "line showing where it will land" is DIY pointer/rect-midpoint math, the same pattern visible inline in the official story's `onDragOver` ([discussion #789](https://github.com/clauderic/dnd-kit/discussions/789)).

**License.** MIT, confirmed at release tags and via npm registry `license` field for both packages.

**Gotchas.** Performance concerns with large non-virtualized lists ([issue #943](https://github.com/clauderic/dnd-kit/issues/943), [#1379](https://github.com/clauderic/dnd-kit/issues/1379)); animation jank/flicker reproducible in the official Storybook itself ([#799](https://github.com/clauderic/dnd-kit/issues/799), [#1950](https://github.com/clauderic/dnd-kit/issues/1950), [#109](https://github.com/clauderic/dnd-kit/issues/109)); the default `rectSortingStrategy` (needed for grid/column layouts) explicitly does not support virtualization, only the vertical/horizontal list strategies do; multi-container edge cases (state freezing, empty-container drops, "too many re-renders" React guard trips) are recurring issue themes ([#386](https://github.com/clauderic/dnd-kit/issues/386), [#828](https://github.com/clauderic/dnd-kit/issues/828), [#1421](https://github.com/clauderic/dnd-kit/issues/1421)). No canonical "Known Limitations" doc page exists; and most importantly, **the classic API is in wind-down** with all forward development on a pre-1.0, API-incompatible successor.

## Native HTML5 Drag and Drop API

**Status.** Not a library - a living part of the [WHATWG HTML spec §6.11](https://html.spec.whatwg.org/multipage/dnd.html), documented primarily by [MDN](https://developer.mozilla.org/en-US/docs/Web/API/HTML_Drag_and_Drop_API). Desktop support is universal and mature (Chrome 4+, Firefox 3.5+, Safari 3.1+, Edge 12+/18+ full). Mobile is the weak spot: iOS Safari had **zero support from 3.2 through 14.8**, only gaining it at iOS 15 (2021) ([caniuse.com/dragndrop](https://caniuse.com/dragndrop)); community polyfills like [`mobile-drag-drop`](https://github.com/timruffles/mobile-drag-drop) exist specifically to paper over this history.

**React 19.** No peerDependency question - wired via `onDragStart`/`onDragOver`/`onDrop`/etc. JSX props delivering a `DragEvent`-shaped SyntheticEvent. No React-19-specific drag behavior changes found; the removal of synthetic event pooling (since React 17) means `event.dataTransfer` can be read asynchronously without `e.persist()`.

**Accessibility.** Essentially none. MDN: _"For the entire course of the drag operation, all device input events (such as mouse or keyboard) are suppressed."_ The spec describes a keyboard model (tab → "copy" → tab → "paste") but **no browser implements it**. There's no dedicated pattern in the [ARIA Authoring Practices Guide](https://www.w3.org/WAI/ARIA/apg/patterns/) for drag-and-drop; `aria-grabbed`/`aria-dropeffect` were deprecated in ARIA 1.1 on an assumption that never materialized. Treat this as **not accessible out of the box** - a parallel keyboard "grab and move" interaction must be hand-built alongside it.

**Touch/pointer.** Confirmed poor historically; `dragstart`/`dragover`/`drop` largely don't fire from touch gestures on many mobile browsers even post-iOS-15. react-dnd's own docs cite this as the reason its `HTML5Backend` needs a separate `react-dnd-touch-backend`.

**Bundle size.** Zero - no JS ships for the API itself.

**Fit for kanban / fractional-rank model.** Naturally index/event-based, since you compute source/destination/insertion index yourself from `dataTransfer` payloads and `getBoundingClientRect()` comparisons - there's no library, so there is nothing to fight for a fractional-rank design. The cost is that **everything is manual**: `preventDefault()` must be called in `dragover` or `drop` never fires; `dataTransfer` is write-only in `dragstart` and read-only in `drop`; Firefox requires `setData()` to be called in `dragstart` or the drag sequence silently never completes ([Bugzilla #1352852](https://bugzilla.mozilla.org/show_bug.cgi?id=1352852)); `setDragImage()` behaves inconsistently across Chrome/Firefox/Safari, including a documented WebKit limitation on overriding default drag graphics ([WebKit bug 24734](https://bugs.webkit.org/show_bug.cgi?id=24734)) and both Firefox/WebKit rasterizing the drag image only after `dragstart` returns ([Bugzilla #692410](https://bugzilla.mozilla.org/show_bug.cgi?id=692410)). No built-in drop-indicator - the entire snap-on-hover visual must be hand-built from `dragover` position math.

**Licensing.** N/A - web platform standard.

**Gotchas.** Summarized above: no mobile support for most of its life, no keyboard/screen-reader model implemented anywhere, Firefox `setData()` requirement, `setDragImage` cross-browser inconsistency, nested-draggable text-selection interference, and the single most common bug ("drop never fires") being a missing `preventDefault()` in `dragover`.

## Other contenders

**react-beautiful-dnd** - dead. Archived by Atlassian, last push 2025-08-18. README states verbatim: _"This project is now archived and is deprecated on npm... To see our ongoing work in the drag and drop problem space, head to Pragmatic drag and drop."_ ([README](https://github.com/atlassian/react-beautiful-dnd/blob/master/README.md)). Last version `13.1.1` (2022-08-30), peer capped at React 18. Dismissed - explicitly superseded by pragmatic-drag-and-drop, no path to React 19.

**@hello-pangea/dnd** - the community-maintained fork continuing react-beautiful-dnd's API after Atlassian's archival. Latest `18.0.1` (2025-02-09), peer `react: "^18.0.0 || ^19.0.0"` (confirmed React 19 support) ([registry](https://registry.npmjs.org/@hello-pangea/dnd/latest)). Actively maintained with steady cadence. Architecturally it inherits rbd's model: `onDragEnd(result)` gives `source`/`destination` indices, but the [docs](https://github.com/hello-pangea/dnd/blob/main/docs/guides/responders.md) assume you reorder your own array the same way rbd always did - functionally index/event-based like the others, but adds no architectural advantage over pragmatic-drag-and-drop or dnd-kit for this project, just a different (single-list-oriented) API surface and Apache-2.0 license. A reasonable fallback if migrating an existing rbd codebase; not a reason to prefer it over the two primary candidates for a greenfield board.

**@formkit/drag-and-drop** - a lightweight, framework-agnostic library with a thin `@formkit/drag-and-drop/react` entrypoint. Latest `0.6.1` (2026-06-15). No declared React peerDependency (works with React 19, uncertified). Bundle: 48.9KB min / 12.6KB gzip for the full package including React wrapper and plugins ([bundlephobia](https://bundlephobia.com/package/@formkit/drag-and-drop@0.6.1)); FormKit's own marketing claims ~4KB gzip for the bare core. Genuinely index/event-based (`onSort` fires with `event.position`, `onTransfer` with `event.targetIndex`) and ships an official kanban-style "transfer" example ([docs/examples/transfer](https://github.com/formkit/drag-and-drop/tree/main/docs/examples/transfer)). Accessibility is fully DIY - FormKit removed an earlier auto-a11y layer in a breaking 0.5.1 change because it didn't generalize. Credible but young (0.x, recent breaking changes) - worth a bookmark, not a first choice given its immaturity relative to the two primary candidates.

**react-dnd** - legacy, stagnant. Latest `16.0.1` (2022-04-19), no release in 4+ years despite sporadic maintenance commits ([commit log](https://github.com/react-dnd/react-dnd/commits/main)). Peer `react: ">= 16.14"` (no upper bound, untested against 19). Architecturally a low-level primitive - index/event-based by design, no built-in sortable/kanban component, and the docs literally instruct you to reorder your own data in the `hover` handler. No accessibility built in. Dismissed as a maintenance risk given its release cadence, not because the architecture is wrong.

**Others checked and ruled out**: `react-sortable-hoc` (README: "no longer actively maintained," ~5 years since last publish); `react-arborist` (tree-view wrapper around react-dnd, not a kanban tool); `Gridstack.js` (actively maintained but an x/y/w/h dashboard-grid engine, not an index/list model, poor fit). shadcn/ui has no official kanban block - every "shadcn kanban" example found (Kibo UI, ReUI, community blocks) wraps **@dnd-kit**, reinforcing it as the 2026 ecosystem default rather than surfacing a new engine.

## Recommendation

### #1: @atlaskit/pragmatic-drag-and-drop

Best fit for the stated requirements. It is the only candidate confirmed **purely index/event-based with zero rendering-time array ownership** - no context/provider anywhere in its API requires a live, ordered id array to render, unlike dnd-kit's `SortableContext`. The official board example demonstrates all three required interactions (cross-column drag, within-column reorder, column-header reorder) using exactly the shape this project needs: `source`, `location.current.dropTargets`, and `extractClosestEdge` feed a pure `getReorderDestinationIndex` function, and the app's own callback decides what "reorder" means - trivially swappable for fractional-rank computation instead of array splicing. It also ships a genuine out-of-the-box snap-on-hover drop-indicator component (`DropIndicator`), which none of the alternatives provide natively. Tradeoffs to accept deliberately: accessibility (keyboard drag, screen-reader announcements) is opt-in and requires real integration work via the `react-accessibility` and `live-region` packages, and touch reliability has open community complaints. Given the project is desktop-first, these are acceptable "add when needed" gaps rather than blockers - and Atlassian's own guidance actively favors an accessible action-menu pattern over faux keyboard-drag anyway.

### #2: @dnd-kit/core + @dnd-kit/sortable

Best accessibility story available (built-in keyboard sensor and live-region announcements, confirmed in official docs) and works fine on React 19 today via the open-ended `>=16.8.0` peer range. The two things holding it out of the #1 slot: (1) `SortableContext` genuinely requires an ordered `items` id array as a render input - workable via deriving it from rank-sorted data each render, but real extra coupling pragmatic-drag-and-drop simply doesn't have; and (2) the classic API (`core`/`sortable`) has been **frozen since December 2024** with the maintainer explicitly redirecting React-19 bug reports to a still-pre-1.0 rewrite (`@dnd-kit/react`) that would require a full migration later. Choose this only if built-in keyboard/screen-reader support is a hard, immediate requirement worth the future migration risk and the array-derivation workaround.

### #3: Native HTML5 Drag and Drop API

The "remove it" rung of the ladder - zero bundle size, zero dependency risk, and naturally index/event-based since there's no library to fight. It's a legitimate choice for a desktop-only board where the team is willing to hand-build the drop-indicator visuals and a parallel keyboard-accessible interaction. It drops below both libraries here because doing so means re-implementing, from scratch and with known cross-browser quirks (Firefox's `setData()` requirement, inconsistent `setDragImage` behavior), exactly the plumbing pragmatic-drag-and-drop already provides for free - not a net simplification once snap-on-hover and any accessibility are actually required, which this project's requirements explicitly call for.

**Not recommended for this project**: react-beautiful-dnd (archived), react-dnd (4+ years stagnant), @hello-pangea/dnd (fine but architecturally redundant with #1/#2, no advantage for a greenfield fractional-rank design), @formkit/drag-and-drop (credible but too immature/0.x to bet a production kanban feature on today).
